WebReg Revamp

Implemented survey method of user research to create a high-fidelity prototype of an improved class registration system.



Insightful ten-week experience working with a team of four where we were taught valuable knowledge on what goes into projects consisting of human-computer interaction. From improving teamwork skills to having the opportunity to complete UX research and usability testing to presenting a presentation, it has been a well-rounded learning experience with many lessons learned.

Purpose | Motivation: The University of California, Irvine’s class registration system, WebReg, is cumbersome to use and requires the necessary use of multiple supplemental tools that add unnecessary complications and increase the risk of errors. If a system relies on multiple supplemental tools, then its usability and convenience are poor and it may benefit from innovation. WebReg’s negative reputation and students’ persistent grievances have created the basis for the motivation of this project.

Design Question: How do we help UCI students with error prevention within the WebReg class registration system, while maintaining the consistency and standards of the current UC standard layout, during class registration periods?

Research Questions: (1) How do the physical layout and aesthetic of the website also affect the UCI student experience? (2) How do other universities handle their class registration? (3) What are the current frustrations, hurdles, or challenges that UCI students face while using WebReg? (4) What features make a class registration system hard to use?

UX Research Methods: Developed methods based on participants that met inclusion criteria, recruitment procedures using Slack and in-person recruitment, data collection procedures involving ensuring integrity and using Google Forms, data analysis plan in analyzing the qualitative and quantitative data, ethical considerations of privacy of our participants, and having surveys as our method of choice rather than interviews to better allow us to ask multiple types of questions to address each research question in order to obtain multiple types of survey data to analyze.

UX Research Results: Quantitative was gathered through graphs and charts. Qualitative data: No search feature. Required to navigate multiple menus. Outdated interface. Server down when high traffic. Hard to view the enrollment window. Hard to visualize study list.

Process: The main needs we addressed in our prototype were directed towards (1) Adding and dropping classes (2) Improving visual of schedule (3) Adjusting waitlist view (4) New menus, split-screen approach

Usability Testing + Competitive Analysis = Conclusion: Surveys gave a greater perspective. Support what works, change what doesn't. Add what's demanded. A complete overhaul of the current website was not necessary. User testing refines changes.


Technologies: Figma, Google Forms, Excel


Course: IN4MATX 132 Project in Human Computer Interaction

Search Engine

Developed an optimized search engine through the process of web crawling, text processing, and ranking weights.




Developed an optimized search engine through the process of web crawling, text processing, term-document matrices, inverted index, index construction, query processing, distributed indexing, ranked retrieval, term weighting, and link analysis. The index was created by obtaining urls of each set of the given data folder and obtaining all the tokens (bold/strong, title, h1, h2, h3) as well as the word count of each document. The index function indexes each token using the url, filepath, index dictionary, token tags, and document id number into 36 pickle files. A report will be returned containing the number of documents, number of unique tokens, file size, and pickle time. The interface will allow users to enter a query and then return results for the top five urls that matched that query.


Technologies: Python


Course: IN4MATX 141 Information Retrieval
